Git flow: как правильно работать с ветками

Хаотичные коммиты прямо в main — это рецепт боли при командной работе. Разбираем простую стратегию ветвления, которая работает как в соло-проектах, так и в команде.

Зачем вообще нужны ветки

Ветка — это изолированная копия кода. Ты делаешь новую фичу в своей ветке, не трогая рабочий код в main. Если что-то пошло не так — просто удаляешь ветку, не засоряя историю.

Основные ветки

main (или master)

Главная ветка. Здесь всегда рабочий, продакшн-готовый код. Никогда не коммить сюда напрямую — только через merge или PR.

develop

Ветка для разработки. В неё вливаются все завершённые фичи. Когда накопился релиз — mergим в main. Используется в командах, для соло-проектов часто не нужна.

Как называть ветки

Чёткое именование — это самодокументация. Рекомендуемый формат:

# Новая фича
feature/user-authentication
feature/dark-mode

# Исправление бага
fix/login-redirect
fix/mobile-menu

# Срочный хотфикс в продакшн
hotfix/payment-crash

# Рефакторинг
refactor/api-layer

Типичный рабочий процесс

# 1. Синхронизируемся с main
git checkout main
git pull origin main

# 2. Создаём ветку для новой фичи
git checkout -b feature/new-navbar

# 3. Работаем, коммитим
git add .
git commit -m "feat: add responsive navbar"

# 4. Пушим ветку
git push -u origin feature/new-navbar

# 5. Создаём Pull Request на GitHub
# 6. После ревью — merge в main, удаляем ветку

Merge vs Rebase

Merge

Создаёт merge-коммит, сохраняет полную историю. Хорошо видно, когда и что вливалось. Подходит для публичных веток.

git checkout main
git merge feature/new-navbar

Rebase

Перемещает коммиты фичи поверх main — история становится линейной и чистой. Подходит для локальных веток до пуша.

# Подтянуть изменения из main в свою ветку без merge-коммита
git checkout feature/new-navbar
git rebase main
⚠️

Никогда не делай rebase публичных веток, в которые пушили другие — это перепишет историю и создаст конфликты у всех.

Удаление веток

После мержа ветки лучше удалять — они засоряют список и вводят в заблуждение.

# Удалить локально
git branch -d feature/new-navbar

# Удалить на remote
git push origin --delete feature/new-navbar
💡

На GitHub в настройках репозитория есть опция «Automatically delete head branches» — ветки будут удаляться автоматически после merge PR.
