## Зачем тестировщику смотреть в код

До этого момента ты работал с Claude через браузер — писал промпты из головы, на основе документации и требований. Это хорошо, но у этого подхода есть потолок.

Документация устаревает. Требования неполные. Разработчик добавил валидацию, о которой не написал в задаче. Или забыл добавить — и в коде дыра.

Когда у тебя есть доступ к коду проекта:
- Ты видишь **реальные** валидации, а не описанные в документе
- Ты находишь места, где валидации **нет** — там будут баги
- Ты понимаешь, **что именно** изменилось в новом релизе
- Ты пишешь баг-репорты с указанием **файла и строки** — разработчики это ценят

И главное: **тебе не нужно читать код самому**. Claude Code читает его за тебя и отвечает на человеческом языке.

## Что такое репозиторий

Если ты никогда не работал с кодом — вот простое объяснение.

**Репозиторий** — это папка со всеми файлами проекта. Код, конфиги, тесты — всё в одном месте.

**Git** — система контроля версий. Как история изменений в Google Docs, только для кода. Можно посмотреть кто, когда и что изменил.

**Read-only доступ** — ты можешь смотреть, но не можешь ничего сломать. Спокойно.

Типичная структура:
```text
project/
├── src/              — исходный код
│   ├── controllers/  — логика обработки запросов
│   ├── models/       — структура данных
│   ├── services/     — бизнес-логика
│   └── validators/   — проверки данных ← тут самое интересное для тебя
├── tests/            — автотесты (можно подсмотреть сценарии)
├── docs/             — документация
└── README.md         — описание проекта
```

Тебе не нужно понимать каждый файл. Тебе нужно знать, что **Claude Code может прочитать их все** и ответить на твои вопросы.

## Установка Claude Code

Шаг за шагом, без паники.

### 1. Установи Node.js

Зайди на nodejs.org, скачай LTS-версию, установи как обычную программу (Next → Next → Finish).

### 2. Открой терминал

- **Windows:** нажми Win, набери "cmd" или "PowerShell", открой
- **macOS:** Spotlight (Cmd+Space), набери "Terminal"

Терминал — это текстовый интерфейс. Ты пишешь команду, нажимаешь Enter, получаешь результат. Ничего страшного.

### 3. Установи Claude Code

Набери в терминале и нажми Enter:

```text
curl -fsSL https://claude.ai/install.sh | bash
```

На Windows (PowerShell):

```text
irm https://claude.ai/install.ps1 | iex
```

Подожди пару минут. Готово. (Старый способ `npm install -g` больше не рекомендуется.)

### 4. Скачай репозиторий

Попроси у разработчика ссылку на репозиторий. Набери:

```text
git clone https://github.com/company/project-name.git
```

Появится папка `project-name` с файлами проекта.

### 5. Запусти Claude Code

```text
cd project-name
claude
```

Всё. Ты внутри проекта, и Claude видит все файлы.

Если что-то не работает — открой claude.ai в браузере и спроси: "Я пытаюсь установить Claude Code, получаю ошибку: [вставь ошибку]. Помоги."

## Расскажи мне про проект

Первое, что ты делаешь после запуска — знакомишься с проектом.

### Общая картина

```text
Расскажи про этот проект. Что это? Какой стек технологий?
Какие основные модули? Объясни простым языком.
```

Claude прочитает файлы проекта и ответит: "Это интернет-магазин на .NET 8 + Angular 19 + PostgreSQL. Основные модули: каталог товаров, корзина, оплата через Stripe, личный кабинет."

### Структура

```text
Покажи структуру папок и объясни, что в каждой папке.
Фокус: где логика, где API, где валидации.
```

### API-эндпоинты

```text
Найди все API-эндпоинты в проекте.
Для каждого покажи: метод, URL, что делает.
```

Теперь ты знаешь все эндпоинты — даже те, которых нет в документации.

### Роли пользователей

```text
Какие роли пользователей есть в системе?
Где проверяются права доступа? Какие эндпоинты доступны каждой роли?
```

## Тест-кейсы из кода

Это ключевое преимущество. Ты генерируешь тест-кейсы не из документации, а из того, что **реально написано в коде**.

### Валидации на форме

```text
Найди все валидации для формы регистрации.
Покажи: какое поле, какое правило, какое сообщение об ошибке.
```

Claude найдёт ВСЕ проверки — и на фронте, и на бэке. Может оказаться, что:
- На фронте email проверяется регуляркой, а на бэке — нет
- Пароль на фронте минимум 6 символов, на бэке — 8
- Поле "телефон" вообще не валидируется

Каждое несоответствие — потенциальный баг.

### Лимиты

```text
Какие лимиты на загрузку файлов в проекте?
Размер, тип, количество. Покажи где это задано в коде.
```

Теперь ты знаешь: максимум 10 МБ, только jpg/png/pdf, до 5 файлов. И знаешь это из кода, а не из документации, которая может врать.

### Сравнение

Тест-кейсы из документации vs из кода:

| Из документации | Из кода |
|-----------------|---------|
| "Пароль минимум 8 символов" | Минимум 8, максимум 128, обязательны: заглавная, цифра, спецсимвол |
| "Email валидируется" | Регулярка на фронте, MX-проверка на бэке, проверка уникальности в БД |
| Не упомянуто | Rate limit: 5 попыток входа, блокировка на 15 минут |

Код не врёт. Документация — может.

## Анализ изменений

### Что изменилось в последнем релизе

```text
Покажи что изменилось за последнюю неделю.
Какие файлы, какие фичи затронуты?
Что может сломаться из-за этих изменений?
```

Claude посмотрит git-историю и скажет: "Изменён модуль оплаты (3 файла), обновлена логика расчёта скидок, добавлен новый API-эндпоинт для экспорта."

Теперь ты знаешь, что тестировать: оплату, скидки, новый эндпоинт + регрессию смежных модулей.

### Risk assessment

```text
Вот список изменённых файлов:
- PaymentService.cs
- DiscountCalculator.cs
- OrderController.cs

Оцени риск каждого изменения.
Что может сломаться? Какие тест-кейсы прогнать?
```

## Поиск проблемных мест

Claude может проанализировать код на типичные проблемы.

### Безопасность

```text
Проверь API-эндпоинты на проблемы безопасности:
- SQL-инъекции
- Отсутствие авторизации
- Доступ к чужим данным (IDOR)
- XSS в ответах
```

### Обработка ошибок

```text
Найди места в коде, где нет обработки ошибок.
Где может вылететь 500-ошибка из-за необработанного исключения?
```

Каждое такое место — готовый тест-кейс.

### Отсутствие валидации

```text
Какие входные данные НЕ валидируются?
Покажи поля/параметры, которые принимаются без проверки.
```

Нет валидации = можно отправить что угодно = баг.

## Понимание ошибок через код

### Стектрейс → точная причина

Раньше ты вставлял стектрейс в Claude и получал общее объяснение. Теперь Claude видит код и может показать **точную строку**.

```text
При тестировании получил ошибку:

NullReferenceException at OrderService.cs:line 47

Покажи этот файл, строку 47 и объясни причину.
Какие данные могли вызвать эту ошибку?
```

Claude откроет файл, покажет строку и скажет: "На строке 47 обращение к order.Customer.Address, но Customer может быть null если пользователь удалил аккаунт. Проверь сценарий: создать заказ → удалить аккаунт → попробовать оплатить."

### Баг-репорт с файлом и строкой

Теперь твой баг-репорт выглядит так:

**Заголовок:** NullReferenceException при оплате заказа удалённого пользователя

**Предположительная причина:** OrderService.cs:47 — нет проверки на null для Customer

Разработчик получает баг и сразу знает где искать. Ты экономишь ему час работы.

> Теперь ты не просто тестировщик — ты тестировщик, который понимает систему изнутри. В финальной части — полный рабочий цикл с Claude, свои скиллы и интеграция с инструментами.
