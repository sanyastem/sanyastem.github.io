## Полный рабочий день с Claude

Пройдём типичный день QA-инженера и покажем, где Claude экономит время.

### 09:00 — Утро: планирование

Открываешь Jira, видишь задачи спринта. Копируешь их в Claude:

```text
Вот задачи спринта 15:
1. SHOP-234: Добавить фильтр по размеру одежды
2. SHOP-235: Исправить баг с двойным списанием при оплате картой
3. SHOP-236: Новая страница "Возврат товара"
4. SHOP-237: Оптимизация загрузки каталога

Составь план тестирования на неделю.
Расставь приоритет по риску для бизнеса.
Для каждой задачи: что тестировать, сколько времени, какие виды тестирования.
```

Через минуту у тебя план: сначала баг с оплатой (critical), потом новая страница, потом фильтр, оптимизацию — в конце.

### 10:00 — Получили PR: анализ

Разработчик закрыл задачу SHOP-234 (фильтр по размеру). Если у тебя есть доступ к репо:

```text
Посмотри последний коммит. Что изменилось?
Какие фичи затронуты? Что может сломаться?
Сгенерируй тест-кейсы на основе изменений.
```

Если нет доступа к репо — используй описание PR:

```text
Вот описание PR:
- Добавлен фильтр по размеру (XS, S, M, L, XL, XXL)
- Фильтр комбинируется с фильтром по цене
- При отсутствии товаров выбранного размера — сообщение "Ничего не найдено"
- Размеры подгружаются из API

Напиши тест-кейсы. Включи: функциональные, UI, граничные, негативные.
```

### 11:30 — Тестируем: нужны данные

Тебе нужны тестовые товары с разными размерами:

```text
Сгенерируй JSON с 20 товарами для тестирования фильтра по размеру.
Поля: id, name, price, sizes (массив), category, in_stock.

Включи:
- Товары со всеми размерами
- Товар с одним размером
- Товар без размеров (пустой массив)
- Товар с ценой 0
- Товар не в наличии
```

### 14:00 — Нашли баг: оформляем

Во время тестирования нашёл: при комбинации фильтра по размеру + цене, если сначала выбрать размер, потом цену, а потом убрать размер — список пустой.

```text
Оформи баг-репорт для Jira:

Фильтр товаров. Выбираю размер M → отображаются товары.
Добавляю фильтр по цене (100-500) → товары фильтруются.
Убираю фильтр по размеру → список пустой, хотя должны остаться
товары в диапазоне 100-500 без фильтра по размеру.
Chrome 125, staging.
```

Скопировал результат в Jira — баг оформлен за 30 секунд.

### 16:00 — Регрессия

Перед релизом нужно прогнать регрессию:

```text
Вот что вошло в релиз 2.15:
- Фильтр по размеру (новая фича)
- Исправление двойного списания
- Новая страница возврата
- Оптимизация каталога

Модули системы: каталог, корзина, оплата, профиль, заказы.

Создай приоритизированный чеклист регрессии.
Время на всё: 3 часа.
```

### 17:30 — Отчёт

```text
Напиши отчёт о тестировании для лида. Кратко, 5-7 предложений.

Данные:
- Протестировано: SHOP-234, SHOP-235, SHOP-236, SHOP-237
- Выполнено тест-кейсов: 67
- Найдено багов: 4 (1 major, 3 minor)
- Major: комбинация фильтров ломает список (SHOP-234)
- Блокеров нет
- Регрессия: без проблем
- Рекомендация: можно релизить после фикса major бага
```

Готово. Весь день — с Claude.

## CLAUDE.md для QA-проекта

Если ты используешь Claude Code с репозиторием (часть 4), создай файл `CLAUDE.md` в корне проекта. Claude будет читать его каждый раз и понимать контекст.

Пример для тестировщика:

```markdown
# E-Shop — интернет-магазин одежды

## Стек
- Backend: .NET 8 Web API
- Frontend: Angular 19
- Database: PostgreSQL 16
- Payments: Stripe
- Hosting: Azure

## Окружения
- Dev: https://dev.eshop.example.com
- Staging: https://staging.eshop.example.com
- Prod: https://eshop.example.com

## Тестовые учётки
- Покупатель: buyer@test.com / BuyerTest123!
- Админ: admin@test.com / AdminTest123!
- Менеджер: manager@test.com / ManagerTest123!

## Бизнес-правила
- Минимальная сумма заказа: 500 руб
- Бесплатная доставка от 3000 руб
- Максимум 10 товаров в корзине
- Возврат в течение 14 дней
- Оплата: карта (Stripe), наличные курьеру

## Критичные модули (тестировать тщательно)
- Оформление заказа + оплата
- Регистрация / авторизация
- Корзина (расчёт суммы, скидки)
- Каталог (фильтры, поиск)

## Известные ограничения
- Safari < 16: некорректное отображение дат
- IE не поддерживается
- Загрузка фото: max 5 МБ, jpg/png
```

Теперь каждый раз, когда ты спрашиваешь Claude Code о проекте, он уже знает контекст.

## Свои скиллы

Скиллы — это сохранённые команды для Claude Code. Набираешь `/qa-testcases` вместо длинного промпта.

### Как создать

В папке проекта создай:

```text
.claude/skills/qa-testcases/SKILL.md
```

Содержимое файла:

```markdown
---
name: qa-testcases
description: Генерация тест-кейсов из требований или кода
allowed-tools: Read, Grep, Glob
---

Сгенерируй тест-кейсы на основе переданного описания
или файла.

Включи: позитивные, негативные, граничные, безопасность.

Формат: таблица с колонками:
ID | Тип | Название | Шаги | Ожидаемый результат | Приоритет
```

Теперь набираешь `/qa-testcases форма оплаты` — и Claude генерирует тест-кейсы.

### Полезные скиллы для QA

**`/qa-bugreport`** — оформляет баг из описания:

```markdown
---
name: qa-bugreport
description: Баг-репорт из свободного описания
allowed-tools: Read, Grep, Glob
---

Оформи баг-репорт из описания.
Формат: заголовок, severity, шаги, expected/actual,
окружение, workaround.
```

**`/qa-checklist`** — чеклист тестирования:

```markdown
---
name: qa-checklist
description: Чеклист тестирования для фичи
allowed-tools: Read, Grep, Glob
---

Создай чеклист тестирования.
Категории: функционал, валидация, UI/UX, безопасность,
производительность, совместимость.
Формат: чекбоксы.
```

Один раз создал — используешь каждый день.

## Интеграция с инструментами

### Claude + Postman

```text
Вот swagger-документация API (или вставь JSON):

Сгенерируй коллекцию для Postman:
- Для каждого эндпоинта: валидный запрос + 3 невалидных
- Pre-request script для авторизации
- Тесты для проверки ответов (status code, обязательные поля)

Формат: JSON для импорта в Postman.
```

### Claude + Jira

```text
Вот результаты тестирования (список тест-кейсов с результатами).

Создай баг-репорты для каждого failed тест-кейса.
Формат: Jira markdown.
Включи: заголовок, severity, шаги, expected/actual.
```

```text
Вот описание фичи. Напиши acceptance criteria для Jira-тикета.
Формат: список чекбоксов (Jira checkbox format).
```

### Claude + TestRail

```text
Вот тест-кейсы (таблица).

Переформатируй для импорта в TestRail:
- Section: [название раздела]
- Title, Preconditions, Steps, Expected Result
- Priority: 1-Critical, 2-High, 3-Medium, 4-Low
- Type: Functional / Regression / Smoke

Формат: CSV для импорта.
```

### Claude + Excel/CSV

```text
Вот данные из баг-трекера за 3 месяца (вставь или опиши).

Создай сводку:
- Количество багов по severity (таблица)
- Количество по модулю
- Среднее время исправления
- Тренд по неделям

Формат: CSV, чтобы я мог вставить в Excel и построить графики.
```

## Чего ты достиг

Если ты прошёл все 5 частей серии, теперь ты можешь:

- Генерировать тест-кейсы за **минуты**, а не часы
- Оформлять баг-репорты за **30 секунд** вместо 10 минут
- Писать тест-планы, регрессии и чеклисты — не с нуля
- Анализировать код проекта, **не зная программирования**
- Находить баги, которые **другие пропускают** — потому что ты видишь код
- Писать баг-репорты с указанием **файла и строки** — разработчики ценят
- Готовить отчёты для менеджера за **2 минуты**

Ты не стал программистом. Ты стал тестировщиком, который **понимает систему изнутри** и использует AI как инструмент.

> Серия завершена. Ты прошёл путь от первого промпта в браузере до полного рабочего цикла с Claude Code. Начни с малого — одна команда в день. Через неделю не сможешь без этого работать.
