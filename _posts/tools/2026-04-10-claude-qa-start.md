## Зачем тестировщику AI-ассистент

Если ты мануальный тестировщик, большая часть твоей работы — это текст. Тест-кейсы, чеклисты, баг-репорты, тестовые данные, уточняющие вопросы к разработчикам. По разным оценкам, 60% рабочего времени QA уходит на документирование, и только 40% — на само тестирование.

Claude берёт на себя текстовую рутину. Ты описываешь задачу — он генерирует структурированный результат. Не идеальный, но на 80% готовый. Тебе остаётся проверить, дополнить контекстом проекта и отправить.

Это не замена тестировщику. AI не знает твой продукт, не видит экран, не понимает бизнес-логику. Но он умеет:

- Быстро генерировать тест-кейсы по описанию фичи
- Оформлять баг-репорты из сумбурных заметок
- Придумывать тестовые данные для граничных случаев
- Находить пробелы в требованиях
- Объяснять непонятные ошибки и логи

Ты тестировщик, и код тебе не нужен. Это нормально. Claude работает через обычный чат.

## Как начать

### claude.ai — через браузер

Самый простой способ. Открываешь [claude.ai](https://claude.ai), регистрируешься через Google-аккаунт или email. Всё, можно работать.

Бесплатный тариф даёт ограниченное количество сообщений в день — для старта хватит. Если понравится, Pro-подписка снимает лимиты и даёт доступ к самой мощной модели.

Что нужно знать:

- **Новый чат** — каждый раз чистый контекст. Claude не помнит прошлые диалоги
- **Длинный контекст** — можно вставлять целые ТЗ, user stories, логи ошибок
- **Projects** — можно загрузить файлы проекта (ТЗ, спецификации) и Claude будет их учитывать в каждом диалоге

### Десктопное приложение

Claude есть как приложение для macOS и Windows. Скачиваешь с [claude.ai/download](https://claude.ai/download). Работает так же, как в браузере, но удобнее переключаться между окнами.

### Что НЕ нужно

- Терминал — не нужен
- Установка пакетов — не нужна
- Знание языков программирования — не нужно
- Ключи API — не нужны

Просто браузер и клавиатура. Если ты умеешь писать в мессенджер — ты умеешь работать с Claude.

> Есть ещё Claude Code — это инструмент для разработчиков, который работает в терминале и умеет редактировать файлы в проекте. Мы доберёмся до него в частях 4-5, когда будем говорить про автоматизацию. Пока забудь о нём.

## Контекст — главное правило

Самое важное, что нужно понять: **чем больше контекста ты даёшь, тем лучше результат**. Claude не видит твой проект, не знает твою команду, не читал ТЗ. Всё, что он знает — это то, что ты написал в сообщении.

### Формула хорошего промпта

Запомни шаблон:

```text
Я тестирую [что именно].
Платформа: [web / mobile / desktop].
Пользователь: [роль].
Нужно: [конкретный результат].
```

Этот шаблон работает для 90% задач. Подставляешь свои данные — получаешь релевантный результат.

### Плохой промпт vs хороший промпт

Плохо:

```text
напиши тест-кейсы для логина
```

Claude выдаст что-то общее и шаблонное. Тест-кейсы для «какого-то логина» без деталей — бесполезная генерика.

Хорошо:

```text
Я тестирую форму авторизации в интернет-магазине.
Web-приложение, десктоп браузер.
Пользователь: покупатель.

Поля:
- Email (обязательное)
- Пароль (обязательное, минимум 8 символов)

Кнопки:
- «Войти»
- «Забыл пароль» — ведёт на /reset-password
- «Регистрация» — ведёт на /register

После 5 неудачных попыток аккаунт блокируется на 30 минут.

Напиши тест-кейсы: позитивные, негативные, граничные значения.
Формат: ID | Название | Шаги | Ожидаемый результат.
```

Разница — как между вопросом «расскажи про погоду» и «какая погода завтра в Минске, нужна ли куртка?». Конкретный вопрос — конкретный ответ.

### Что включать в контекст

- **Что тестируешь** — название фичи, модуля, экрана
- **Бизнес-правила** — ограничения, лимиты, роли пользователей
- **Технические детали** — платформа, браузер, версия API
- **Формат ответа** — таблица, Gherkin, чеклист, свободный текст
- **Scope** — что покрыть: только happy path или граничные случаи тоже

## 5 первых команд тестировщика

Вот пять задач, которые ты можешь отдать Claude прямо сейчас. Каждую — с конкретным промптом.

### 1. Тест-кейсы для формы регистрации

```text
Я тестирую форму регистрации в мобильном приложении банка.
Платформа: iOS и Android.
Пользователь: новый клиент.

Поля:
- Имя (2–50 символов, только буквы и дефис)
- Телефон (формат +375XXXXXXXXX)
- Email (опционально)
- Пароль (мин. 8 символов, заглавная, цифра, спецсимвол)
- Повторите пароль

Есть чекбокс согласия с офертой (обязательный).
Есть CAPTCHA после 3 неудачных попыток.

Напиши тест-кейсы: позитивные, негативные, граничные.
Формат: таблица с колонками ID, Название, Предусловия, Шаги, Ожидаемый результат.
```

Claude вернёт таблицу с 20-30 кейсами. Там будут проверки валидации каждого поля, граничные длины, несовпадение паролей, пустые обязательные поля, спецсимволы в имени. Твоя задача — добавить кейсы, специфичные для твоего банка (например, проверка телефона через SMS).

### 2. Баг-репорт из свободного описания

```text
Оформи баг-репорт из моего описания.

Вот что произошло: зашёл в корзину, нажал «Оформить заказ»,
выбрал доставку курьером, ввёл адрес, нажал «Далее».
Страница повисла на 10 секунд, потом показала ошибку
«Не удалось рассчитать стоимость доставки».
При этом самовывоз работает нормально.
Баг появился после вчерашнего релиза.
Браузер: Chrome 124, Windows 11.
Окружение: staging.

Формат:
- Заголовок
- Окружение
- Предусловия
- Шаги воспроизведения
- Фактический результат
- Ожидаемый результат
- Severity / Priority с обоснованием
```

Claude структурирует твои заметки в аккуратный баг-репорт. Он предложит Severity (скорее всего Major — блокирует оформление заказа для части пользователей) и Priority. Проверь, совпадает ли с политикой твоей команды.

### 3. Чеклист для тестирования корзины

```text
Составь чеклист для тестирования корзины интернет-магазина.

Функциональность:
- Добавление товара (из каталога и карточки товара)
- Изменение количества (от 1 до 99, не больше остатка на складе)
- Удаление товара
- Промокод (скидка в %, скидка фиксированная, бесплатная доставка)
- Промокод нельзя применить к товарам со скидкой
- Пересчёт суммы при любых изменениях
- Переход к оформлению заказа

Формат: чеклист с группировкой по блокам.
Отметь приоритет каждой проверки: P1 (критично), P2 (важно), P3 (желательно).
```

Получишь структурированный чеклист на 40-50 пунктов, сгруппированный по блокам. Удобно импортировать в TestRail или любую другую TMS. Claude обычно хорошо расставляет приоритеты — P1 для расчёта суммы и оформления, P3 для UI-мелочей.

### 4. Тестовые данные для поля email

```text
Сгенерируй тестовые данные для проверки поля email.
Максимальная длина — 254 символа.
Используется для регистрации в веб-приложении.

Нужны:
- 5 валидных email (разные форматы: с точкой, с плюсом, с цифрами)
- 10 невалидных email (без @, два @, пробелы, кириллица, спецсимволы, пустая строка)
- 3 граничных случая (максимальная длина, минимальная длина, ровно на границе)

Формат: таблица — Значение | Тип (валидный/невалидный/граничный) | Что проверяет
```

Claude выдаст таблицу с готовыми тестовыми данными. Не нужно придумывать самому email с двумя собаками или проверять, какая максимальная длина у домена. Скопировал — вставил в тест.

### 5. «Объясни эту ошибку»

```text
Я мануальный тестировщик, не разработчик. Объясни простым языком,
что означает эта ошибка и что мне написать в баг-репорте.

Ошибка из консоли браузера:

Uncaught TypeError: Cannot read properties of undefined (reading 'map')
    at ProductList.render (ProductList.jsx:24)
    at renderWithHooks (react-dom.development.js:16305)
    at mountIndeterminateComponent (react-dom.development.js:20074)

На странице каталога вместо списка товаров — пустой экран.
```

Claude объяснит: приложение пытается отобразить список товаров, но данные не пришли с сервера (или пришли в неправильном формате). Компонент `ProductList` на строке 24 ожидает массив, а получил `undefined`. Он подскажет, что написать в баг-репорте, включая ссылку на файл и строку.

Это особенно полезно, когда разработчик скинул лог и спросил «а ты что видел?» — можешь ответить по существу, а не «я не программист».

## Итерация — второе правило

Первый ответ Claude — это черновик. Не финальный результат. Продолжай диалог:

```text
Хорошо, но добавь ещё кейсы для мультиязычности — 
у нас интерфейс на русском, белорусском и английском.
```

```text
Перепиши в формате Gherkin. Нужно для Cucumber.
```

```text
Убери кейсы с API — я тестирую только через UI.
```

```text
Сгруппируй по приоритету: сначала P1, потом P2.
И добавь колонку «Тестовые данные» к каждому кейсу.
```

Каждое уточнение делает результат ближе к тому, что тебе нужно. Claude помнит весь диалог в рамках одного чата, поэтому не нужно повторять контекст.

Три-четыре итерации — и у тебя готовый артефакт, который можно нести в Jira.

## Организация работы: Projects

В claude.ai есть раздел **Projects**. Это как папка, в которую можно загрузить файлы и написать системную инструкцию. Каждый новый чат внутри проекта автоматически получает этот контекст.

Как использовать для тестирования:

1. Создай проект, назови по имени продукта — например, «QA: Интернет-магазин»
2. Загрузи туда ТЗ, спецификации, user stories в текстовом формате
3. В системной инструкции напиши базовый контекст:

```text
Ты помогаешь мануальному тестировщику. 
Проект: интернет-магазин электроники.
Платформа: web (React), мобильное приложение (React Native).
Пользователи: покупатели, менеджеры, администраторы.
Формат тест-кейсов: ID | Название | Шаги | Ожидаемый результат.
Язык ответов: русский.
```

Теперь в каждом новом чате внутри проекта не нужно повторять «я тестирую интернет-магазин, платформа web...» — Claude уже знает.

Можно создать отдельные проекты для разных продуктов или для разных типов задач (один проект для тест-кейсов, другой для баг-репортов).

## Чего Claude не может

Прежде чем ты начнёшь делегировать всё подряд, вот границы:

### Не видит экран

Claude работает только с текстом. Он не может нажать кнопку, открыть браузер, проскроллить страницу. Ручное тестирование — это по-прежнему ты.

Можно вставить скриншот в чат (claude.ai поддерживает изображения), и Claude опишет, что видит. Но это не замена реальному прохождению тест-кейса.

### Не знает бизнес-контекст

Claude не знает, что в твоём проекте промокод `WELCOME10` даёт скидку только новым пользователям, а `VIP30` — только для пользователей с историей покупок больше 50 000 рублей. Если не расскажешь — он не учтёт.

Всегда добавляй бизнес-правила в промпт. Чем подробнее, тем лучше.

### Может ошибаться

Claude уверенно генерирует тест-кейсы, но может:

- Придумать несуществующие ограничения
- Пропустить важный сценарий
- Неправильно оценить severity
- Предложить невозможный шаг воспроизведения

Каждый результат нужно проверять. Это инструмент для ускорения, а не для слепого доверия.

### Не заменяет твою экспертизу

Ты знаешь, где в приложении «тонкие места». Знаешь, какие баги уже были и могут вернуться. Знаешь, какой флоу пользователи используют чаще всего. AI этого не знает.

Лучшая схема: Claude генерирует 80% — ты добавляешь 20% из своего опыта.

### Конфиденциальность

Не вставляй в Claude реальные данные пользователей, пароли, токены, ключи API. Если нужно показать структуру данных — замени реальные значения на фейковые. Вместо настоящего email клиента пиши `test.user@example.com`. Это правило безопасности, не рекомендация.

## Шпаргалка

Краткий чеклист для начала работы:

1. Открой [claude.ai](https://claude.ai) и зарегистрируйся
2. Используй формулу контекста: **что тестируешь + платформа + пользователь + что нужно**
3. Начинай с конкретной задачи — тест-кейсы для фичи, которую сейчас тестируешь
4. Итерируй — уточняй, добавляй детали, меняй формат
5. Создай Project для своего продукта, загрузи ТЗ
6. Всегда проверяй результат — Claude ассистент, не оракул
